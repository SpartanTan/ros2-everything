Nav2 / BehaviorTree 学习对话整理

说明
- 这是根据当前会话整理出的对话记录稿，主题集中在 Nav2、BehaviorTree.CPP、bt_navigator、behavior_server、controller_server、BT 插件与行为插件的关系。
- 为了可读性，内容按问答顺序保留核心原话与结论。

1. clang-format 与 VS Code

用户：我已经安装了 clang-format，我如何确定现在 VS Code 是通过 clang-format 进行格式化而非自带的 C/C++？
答：可通过 `C_Cpp.formatting = clangFormat` 配置和 VS Code Output 面板中的 `C/C++` 日志确认。若日志里出现 `Formatting with clang-format` 或实际的 clang-format 路径，说明使用的是 clang-format。

用户：为什么第 9 行这么长还不换行？
答：因为 style 里设置了 `ColumnLimit: 0`，这会关闭基于列宽的自动换行。要启用自动换行，需要设置如 `ColumnLimit: 100`。

用户：即使我这里写了 .clang-tidy，但是工作空间目录下只有 .clang-format 文件，一样能起作用？
答：不能。`.clang-format` 只用于格式化，`.clang-tidy` 只用于静态检查。两者互不替代。

2. BehaviorTree.CPP 基础概念

用户：`factory.registerNodeType<SaySomething>("SaySomething");` 为什么这里只提供了一个 name？
答：这不是直接构造 `SaySomething` 对象，而是在把类型 `SaySomething` 以注册名 `"SaySomething"` 注册到工厂。真正构造对象发生在工厂解析 XML 并建树时，框架内部才会调用 `SaySomething(name, config)`。

用户：`name="failing_action"` 是输入 port 名吗？
答：不是。这里的 `name` 是节点实例名，主要用于日志、调试、可视化和路径标识，不是 port。

用户：XML 中 `name="check_battery"` 可以随便取吗？
答：通常可以。必须和注册名一致的是 XML 标签名如 `<CheckBattery/>`，不是 `name` 属性。

用户：匿名函数中的 `BT::TreeNode& self` 是函数参数吗？
答：是。它是 lambda 的形参，表示当前正在执行的 BT 节点对象引用。

用户：`registerBehaviorTreeFromText(xml_main);` 和 `createTreeFromFile(...)` 有区别吗？
答：有。前者是“注册树定义”，后者是“从文件读取并立即创建一棵可执行树实例”。

用户：`ReactiveSequence` 和普通 `Sequence` 有什么区别？
答：`ReactiveSequence` 每次 tick 都会从第一个子节点重新检查前置条件；普通 `Sequence` 更偏向在已有 RUNNING 子节点上继续，不会每次都重查前面已成功节点。

用户：`<Precondition if="battery_ok" else="RUNNING">` 中的 `else` 是什么意思？
答：若条件不满足，子节点 `<MoveToGoal/>` 不执行，整个 `Precondition` 节点直接返回 `RUNNING`。

用户：如果 `MoveToGoal` 需要执行一段时间，这段时间会阻塞 tick 吗？
答：取决于节点实现。正确的长时动作应快速返回 `RUNNING` 而不阻塞；如果在 `tick()` 或 `onRunning()` 中直接睡眠或等待，就会阻塞该次 tick。

用户：`reactive_behaviors.cpp` 中每次 `onRunning()` 都 `sleep_for(10ms)`，会不会拖慢整个 tick 节奏？
答：会。这 10ms 是在当前 tick 调用链里发生的，因此会直接拉长当前树 tick 的耗时。

3. Nav2 行为树总体结构

用户：我已经学会了使用 BehaviorTree.CPP，想进一步了解 Nav2 是如何构建和使用行为树的。
答：推荐把 Nav2 分为几层来理解：
- 顶层宿主节点：`bt_navigator`
- navigator 插件：如 `NavigateToPoseNavigator`
- 顶层 action server：如 `navigate_to_pose`
- 行为树 XML
- BT 插件节点：如 `ComputePathToPoseAction`、`FollowPathAction`、`SpinAction`
- 下游 server：`planner_server`、`controller_server`、`behavior_server`
- server 内部算法插件：planner plugin、controller plugin、behavior plugin

典型主链：
外部客户端 -> `navigate_to_pose` action server -> 行为树运行
-> `ComputePathToPose` 生成路径
-> `FollowPath` 发送请求给 `controller_server`
-> `controller_server` 内部控制器插件持续发布 `cmd_vel`

4. bt_navigator / Navigator / BtActionServer

用户：`NavigateToPose` 再上层还有东西吗？`bt_navigator`？再上层还有吗？
答：有。`bt_navigator` 是宿主节点，负责加载 navigator 插件并提供顶层导航 action。再上层还有外部客户端，如 RViz、你的上层任务系统或 mission planner。

用户：`bt_navigator` 和 `BehaviorTreeNavigator` 是什么关系？
答：`bt_navigator` 是具体 ROS2 节点/宿主；`BehaviorTreeNavigator<ActionT>` 是 navigator 插件继承的通用基类框架。实际是 `bt_navigator` 加载 `NavigatorBase` 类型插件，而 `NavigateToPoseNavigator` 继承自 `BehaviorTreeNavigator<nav2_msgs::action::NavigateToPose>`。

用户：`on_configure()` 为什么基类也会被调用？
答：因为 `BehaviorTreeNavigator::on_configure()` 是 `final`，子类并没有 override 它；子类 override 的是 `configure()`。基类 `on_configure()` 负责创建 `BtActionServer<ActionT>` 等通用逻辑，最后再主动调用子类的 `configure()` 扩展点。

用户：从 `bt_navigator` 开始，把 `on_activate()` 这一条通路告诉我。
答：生命周期管理器调用 `BtNavigator::on_activate()`，后者遍历已加载的 navigator 插件，调用 `navigators_[i]->on_activate()`。运行时多态会进入 `BehaviorTreeNavigator<ActionT>::on_activate()`，其中激活 `bt_action_server_`，然后再调用子类的 `activate()` 扩展点。

5. 顶层 NavigateToPose action server

用户：我的理解是，`NavigateToPoseNavigator` 有一个 `BtActionServer`，而 `BtActionServer` 里再持有 `SimpleActionServer`，最终由 `rclcpp_action::create_server<ActionT>()` 创建 server，action name 来自 `getName()` 返回的 `"navigate_to_pose"`，对吗？
答：对。更准确地说：
- `bt_navigator` 加载的是 `NavigatorBase` 插件
- 具体实例是 `NavigateToPoseNavigator`
- 它继承自 `BehaviorTreeNavigator<nav2_msgs::action::NavigateToPose>`
- 基类内部创建 `BtActionServer<nav2_msgs::action::NavigateToPose>`
- `BtActionServer` 内部持有 `SimpleActionServer<nav2_msgs::action::NavigateToPose>`
- `SimpleActionServer` 最终调用 `rclcpp_action::create_server<ActionT>(..., "navigate_to_pose", ...)`

用户：`handle_goal`、`handle_cancel`、`handle_accepted` 在哪里？
答：不在 `NavigateToPoseNavigator` 中，而在 `nav2_util::SimpleActionServer` 中实现。`handle_accepted()` 会启动异步执行线程，最终调用 `BtActionServer::executeCallback()`。

用户：`handle_accepted()` 之后具体做什么？
答：它本身不直接写 BT 逻辑，而是启动异步工作线程。随后 `executeCallback()` 被调用，在其中先执行 `on_goal_received_callback_`，再调用 `bt_->run()` 周期性 tick 行为树。

用户：`navigate_to_pose` 收到 goal 后，行为树是怎么和它联系起来的？
答：`NavigateToPoseNavigator::goalReceived()` 会加载当前要执行的 XML，并把目标位姿写到 blackboard 的 `goal` 键中。随后 `BtActionServer::executeCallback()` 调 `bt_->run()`，树中的 `<ComputePathToPose goal="{goal}" .../>` 会从 blackboard 取这个值。

6. BehaviorTreeEngine 与 BT 插件注册

用户：`bt_ = std::make_unique<nav2_behavior_tree::BehaviorTreeEngine>(plugin_lib_names_, client_node_);` 这一句做了什么？可以说“注册了 BT 插件吗”？
答：可以口语上这么说，但更准确是：
- 这句代码创建了 `BehaviorTreeEngine`
- 真正的插件注册发生在其构造函数里
- 构造函数遍历 `plugin_lib_names_`，调用 `factory_.registerFromPlugin(loader.getOSName(p))`
- 加载共享库后，会执行库内的 `BT_REGISTER_NODES(factory)`，从而把 `FollowPath`、`ComputePathToPose`、`Spin` 等节点注册到 `BehaviorTreeFactory`

用户：`BT_BUILTIN_PLUGINS` 里只有 `nav2_follow_path_action_bt_node`，并没有 `FollowPath`，为什么？
答：因为 `plugin_libraries` 列表保存的是共享库名，而不是 BT 节点 ID。三者是不同层级：
- 插件库名：`nav2_follow_path_action_bt_node`
- C++ 类：`nav2_behavior_tree::FollowPathAction`
- BT XML 节点 ID：`"FollowPath"`
- ROS action 名：`"follow_path"`

7. XML `<FollowPath>` 到 `FollowPathAction`

用户：XML 被解析时，它怎么知道 `<FollowPath ...>` 对应 `FollowPathAction`？
答：因为插件库加载时执行了 `BT_REGISTER_NODES(factory)`，其中有：
`factory.registerBuilder<nav2_behavior_tree::FollowPathAction>("FollowPath", builder);`
也就是建立了 `"FollowPath" -> builder -> FollowPathAction` 的映射。

用户：构造 `FollowPathAction` 时，`xml_tag_name`、`action_name`、`conf` 是在哪里提供的？
答：
- `action_name` 是在 builder 里手动硬编码的 `"follow_path"`
- `name`（构造函数里常被命名为 `xml_tag_name`）和 `config` 都是 BT factory 在解析 XML 并创建节点实例时传给 builder 的

用户：为什么 `xml_tag_name` 这个变量名让我误解？
答：因为它容易让人以为它严格等于 XML 标签类型名 `<FollowPath>` 里的 `FollowPath`。实际上更适合把它理解为“BT 节点实例名/节点名”。命名上确实容易误导。

8. NodeConfiguration / blackboard / port 类型

用户：`config` 中的数据后面在哪里会被用到？
答：主要用于三件事：
- 端口映射：`getInput()` / `setOutput()`
- blackboard 访问：如 `config().blackboard->get<...>("node")`
- manifest / 默认值 / pre/post 条件

用户：为什么 `TreeNode::config()` 在头文件里没有实现代码？
答：因为那只是声明，实现通常在对应 `.cpp` 文件中。`TreeNode` 是普通类，不像模板那样必须把实现都放进头文件。

用户：对于行为树来说，只知道输入输出端口和 `{path}` 变量绑定，但不知道 `path` 是什么类型，对吗？
答：XML 本身不显示类型，但并不是“完全没类型”。类型来自 C++ 中的 `providedPorts()` 声明，以及 `getInput<T>() / setOutput<T>()` 模板参数。这里的 `path` 类型是 `nav_msgs::msg::Path`，不是不存在的 `nav2_msgs::msg::Path`。

9. planner_server 结果如何传到 FollowPath

用户：在 planner server 中，plan 出来的 path 是如何传递到行为树中交给 `FollowPath` 的？
答：通过 `ComputePathToPose` 节点的输出端口和 blackboard 中转：
- XML：`<ComputePathToPose ... path="{path}" .../>`
- `ComputePathToPoseAction::on_success()` 调 `setOutput("path", result_.result->path);`
- blackboard 中的 `path` 被写为 `nav_msgs::msg::Path`
- XML：`<FollowPath path="{path}" .../>`
- `FollowPathAction::getInput("path", goal_.path);`
- 再发送给 `controller_server`

10. FollowPath -> controller_server

用户：`FollowPath` 怎么知道 goal 要发给哪个 controller server？
答：`FollowPathAction` 在 builder 中把 ROS action 名硬编码为 `"follow_path"`。`BtActionNode` 使用这个 action name 创建 action client；而 `controller_server` 创建的 action server 名也正是 `"follow_path"`，二者因此对上。

用户：`controller_id` 是不是 server 名？
答：不是。`controller_id` 是 `FollowPath` action request 中的一个字段，用来告诉 `controller_server` 在其内部已加载的多个 controller plugin 中选用哪一个。真正的 action server 名是 `"follow_path"`。

用户：所以 Nav2 中，`FollowPath` 这个 ActionClient 的名字是 `"follow_path"`，对应的 `controller_server` 这个 ActionServer 的 name 也是 `"follow_path"`，对吗？
答：对。

11. 顶层 RUNNING 状态

用户：当运行到例如 `FollowPath` 之后，它向 `controller_server` 发送 request，控制会花一段时间。这期间由于 `executeCallback()` 在单独线程中运行，因此不会阻塞 `navigate_to_pose`，那么 `navigate_to_pose` 的状态是什么？
答：顶层 `navigate_to_pose` action 通常会一直保持 `RUNNING`。因为：
- `FollowPath` BT 节点在下游 action 未结束前会持续返回 `RUNNING`
- 整棵树保持 `RUNNING`
- `BtActionServer::executeCallback()` 也未返回
直到树最终成功、失败或取消，顶层 action 才结束。

12. 插件类别在 Nav2 中的对应关系

用户：官方文档提到 Behavior Plugin、Behavior Tree Plugin、Navigator Plugin 等等，它们具体对应哪一个模块？
答：
- Navigator Plugin：`nav2_bt_navigator`，基类 `NavigatorBase` / `BehaviorTreeNavigator<ActionT>`，例如 `NavigateToPoseNavigator`、`NavigateThroughPosesNavigator`
- Behavior Tree Plugin：`nav2_behavior_tree/plugins/...`，通过 `BT_REGISTER_NODES(factory)` 注册，例如 `FollowPathAction`、`ComputePathToPoseAction`、`SpinAction`
- Behavior Plugin：`nav2_behaviors/plugins/...`，基类 `nav2_core::Behavior`，由 `behavior_server` 通过 pluginlib 加载，例如 `Spin`、`BackUp`、`DriveOnHeading`、`Wait`

13. behavior_server 与 Behavior Plugin

用户：之前的学习中从来没碰见过 Behavior Plugin，它在哪里被用到了？
答：Behavior Plugin 不是给 `bt_navigator` 直接加载的，而是给 `behavior_server` 加载的。行为树中的某些 BT 节点（如 `SpinAction`、`BackUpAction`、`WaitAction`）通过 action client 发请求给 `behavior_server`，后者再把请求交给内部的 Behavior Plugin 执行。

用户：为什么还需要 `behavior_server`？所有动作都交给 `controller_server` 不行吗？
答：不合适。`controller_server` 的职责是“沿路径连续控制”，而 `behavior_server` 负责“独立行为动作/恢复动作”，例如原地旋转、后退、等待。它们输入语义和算法结构不同，不应该混在一起。

用户：如果 `bt_navigator` 加载 `NavigateToPoseNavigator` 并创建了 `navigate_to_pose` server，这个 `navigate_to_pose` server 和 `behavior_server` 是同一级的吗？如果是的话，`behavior_server` 是谁创建的？
答：是同一级的独立 ROS2 节点。`behavior_server` 不是由 `bt_navigator` 创建，而是由 Nav2 bringup 的 launch 文件直接启动，与 `bt_navigator`、`controller_server`、`planner_server` 并列。

14. launch -> behavior_server -> Spin/Wait 插件加载

用户：带我走一下从 launch 文件开始，启动 `behavior_server` 并加载 `spin`、`wait` 等插件的过程。
答：
- `navigation_launch.py` 中启动 `behavior_server`
- 参数文件 `nav2_params.yaml` 中定义 `behavior_plugins: ["spin", "backup", "drive_on_heading", "assisted_teleop", "wait"]`
- 并配置 `spin.plugin = "nav2_behaviors::Spin"` 等类型
- `behavior_server` 在 `loadBehaviorPlugins()` 中用 `plugin_loader_("nav2_core", "nav2_core::Behavior")` 加载这些类

15. SpinAction 与 nav2_behaviors::Spin

用户：`follow_path_action.cpp/hpp` 实现的是可供行为树调用的入口，真正执行在 `controller_server` 那边。那现在对于 `spin.cpp`，它并非是一个对应 XML 的入口，而是一个插件？这里我理解不了。
答：对。要区分两层：
- `spin_action.cpp/hpp` 是 Behavior Tree Plugin，是 XML `<Spin .../>` 的入口
- `nav2_behaviors/plugins/spin.cpp` 是 Behavior Plugin，是 `behavior_server` 内部真正执行旋转的算法插件

并列结构：
`<FollowPath/> -> FollowPathAction -> controller_server -> controller plugin`
`<Spin/> -> SpinAction -> behavior_server -> nav2_behaviors::Spin`

16. TimedBehavior 与 spin.cpp 的 action server

用户：`spin_action.hpp/cpp` 是 BT plugin；`spin.cpp` 是 behavior_server 的插件，这个插件内部持有 `ActionServer`。有两个问题：第一，`spin.cpp` 中持有的 action server 在哪里被命名为 `"spin"`？第二，它的 `goal_accepted` 实现在哪里？
答：
- `"spin"` 这个 action server 名不是在 `spin.cpp` 里写死，而是 `behavior_server` 在调用插件 `configure()` 时把插件 ID `spin` 作为 name 传进去；`TimedBehavior::configure()` 用这个 name 创建 `SimpleActionServer`
- `goal_accepted` 并不是 `spin.cpp` 自己实现的，而在 `nav2_util::SimpleActionServer` 中；`TimedBehavior` 创建 `SimpleActionServer` 时把执行回调绑定为 `std::bind(&TimedBehavior::execute, this)`
- 随后 `handle_accepted()` 异步触发 `TimedBehavior::execute()`，它先调用子类实现的 `onRun()`，然后循环调用子类实现的 `onCycleUpdate()`

17. NavigateToPose 与 Spin 两条链的对照

用户总结：之前在了解 `NavigateToPose` 时，链路是 `bt_navigator -> NavigateToPose -> action server -> SimpleActionServer::handle_accepted() -> BtActionServer::executeCallback() -> 运行行为树`
现在对于 `spin.cpp`，链路为 `spin.cpp(TimedBehavior) 持有 action server -> SimpleActionServer::handle_accepted() -> TimedBehavior::execute() -> onRun/onCycleUpdate`
答：对。主干结构非常相似，只是：
- `NavigateToPose` 在 accepted 之后运行的是“行为树”
- `Spin` 在 accepted 之后运行的是“具体行为插件逻辑”

18. 最终小结

用户：现在小结一下，也就是说在 Nav2 中，navigator 会创建一个 action server 来处理例如 `NavigateToPose` 这样的 request。然后它收到 request 后开始运行行为树，行为树中运行 `ComputePath` 或 `FollowPath` 这样的插件，这些插件从行为树获取参数，然后通过内部的 action client 发出 request 给 `follow_path` 这样的 action server（即 `controller_server`），后者再调用内部的控制算法插件等发布 `cmd_vel` 命令。我的理解对吗？
答：对，整体理解已经正确。再补一句：
- `NavigateToPose` 是顶层任务 action
- `ComputePathToPose` / `FollowPath` / `Spin` 是行为树内部调用的下游 action
- controller plugin / behavior plugin 不是 action server，而是各自 server 内部的算法插件
